# 文生图功能必要理解

> 本文件整理文生图功能涉及的核心技术议题：Electron 安全模型与资源加载机制、路由策略、图片存储方案、厂商 API 兼容性。

---

## 一、Electron 安全模型与自定义协议

### 1.1 Electron 的架构：主进程 vs 渲染进程

Electron 应用由两种进程构成：

| 进程 | 角色 | 权限 |
|------|------|------|
| **主进程 (Main Process)** | 管理窗口、系统对话框、文件读写 | 完整 Node.js 权限 |
| **渲染进程 (Renderer Process)** | 显示页面 UI | 无 Node.js 权限（沙箱中） |

渲染进程本质就是一个 **Chromium 浏览器标签页**。你看到的窗口确实是 Chrome 内核，但每个标签页运行在沙箱中，不能直接访问本地文件系统。

### 1.2 contextIsolation 与 nodeIntegration

当前项目的安全配置：

```javascript
// main/main.js
webPreferences: {
  preload: path.join(__dirname, 'preload.js'),
  nodeIntegration: false,    // 渲染进程不能 require('fs')
  contextIsolation: true,    // 渲染进程与 preload 隔离
}
```

这意味着：
- 渲染进程不能 `require('fs')` 读写文件
- 渲染进程不能 `require('electron')` 调用系统 API
- 渲染进程只能通过 `preload.js` **桥接**（contextBridge）来有限地与主进程通信
- preload 脚本运行在隔离上下文，它暴露什么 API，渲染进程才能用什么

所以 **即使你的应用跑在 Electron 窗口中，渲染进程和打开一个网页没有本质区别**——它不能直接读写本地磁盘。

### 1.3 渲染进程加载资源的机制

渲染进程要显示一张图片，必须在 `<img>` 标签的 `src` 属性中提供一个 **URL**。Chromium 支持的 URL 协议：

| 协议 | 来源 | 能否加载本地文件 |
|------|------|-----------------|
| `http://` / `https://` | 网络 | 不能（除非有本地 HTTP 服务） |
| `file://` | 本地文件系统 | 可以，但有跨域限制 |
| 自定义协议（`app-img://`） | Electron 注册 | 可以，无跨域限制 |

`<img src="file:///C:/path/image.png">` 看上去能直接加载本地文件，但有两个问题：

1. **跨域限制**：Chromium 将不同路径的 `file://` 视为不同 origin。页面本身通过 `http://localhost:3000`（dev）或 `file:///out/index.html`（prod）加载，再去加载另一个路径的 `file://` 资源，可能被跨域策略拦截
2. **Windows 格式陷阱**：`file://` 在 Windows 上要求 `file:///C:/path` 格式，反斜杠和双斜杠都不识别

### 1.4 自定义协议的原理

**什么时候需要自定义协议：** 当渲染进程需要加载本地文件系统中的资源（图片、视频等），而这些文件不在页面本身的资源目录中时，需要一种机制让渲染进程能 **通过 URL 访问** 这些文件，同时绕过浏览器的跨域限制。

**什么时候不需要：** 纯文本内容（Markdown、JSON 等）直接通过 IPC 传递字符串即可，不涉及 URL 资源加载。

自定义协议的工作方式：

```
渲染进程                          主进程
   │                                │
   │  <img src="app-img:///path">   │
   │ ──────────────请求─────────────>│
   │                                │  protocol.handle 拦截
   │                                │  主进程读文件（有 fs 权限）
   │  <──────────────响应────────────│
   │  显示图片                       │
```

关键点：**渲染进程不需要文件系统权限，它只需要发送一个 URL 请求。** 主进程收到请求后，用自己的完整 Node.js 权限去读文件，然后把数据返回给渲染进程。

### 1.5 注册自定义协议的步骤

两步分别对应 Electron 生命周期中的不同阶段：

**第一步：`protocol.registerSchemesAsPrivileged`**（必须在 `app.whenReady()` **之前**调用）

告诉 Electron 内核："我要注册一个自定义协议，请给它这些特权"。特权包括：
- `bypassCSP`：绕过内容安全策略限制
- `stream`：支持流式响应
- `supportFetchAPI`：支持 fecth 方式请求

**第二步：`protocol.handle`**（在 `app.whenReady()` **之后**调用）

实现实际的请求处理逻辑。handler 接收请求 URL，返回响应数据。

```javascript
// 必须放在 app.whenReady() 之前
protocol.registerSchemesAsPrivileged([
  { scheme: 'app-img', privileges: { bypassCSP: true, stream: true, supportFetchAPI: true } },
]);

app.whenReady().then(() => {
  // 必须放在 app.whenReady() 之后
  protocol.handle('app-img', (request) => {
    // app-img:///E:/path/file.png → file:///E:/path/file.png
    const url = request.url.replace('app-img:', 'file:');
    return net.fetch(url);
  });
  createWindow();
});
```

### 1.6 当前方案：`app-img://` 协议

| 项目 | 值 |
|------|-----|
| 协议名 | `app-img://` |
| URL 格式 | `app-img:///{绝对路径}`（正斜杠） |
| 图片位置 | 开发：`prisma/images/{roleId}/{timestamp}.png`，生产：`userData/images/{roleId}/{timestamp}.png` |
| 存储到 DB | 存 `app-img:///` URL，而非绝对路径 |
| 前端渲染 | `<img src={msg.content}>`（msg.content 已含协议） |

**路径格式处理：**

```
原始路径:  E:\Claude-code\zoe\prisma\images\xxx\123.png
        ↓ replace(/\\/g, '/')
存储 URL:  app-img:///E:/Claude-code/zoe/prisma/images/xxx/123.png
```

**跨平台验证：**

| 平台 | `app-img:///` URL | 协议 handler 转换后 |
|------|-------------------|--------------------|
| Windows | `app-img:///E:/path/file.png` | `file:///E:/path/file.png` |
| macOS | `app-img:///Users/name/file.png` | `file:///Users/name/file.png` |

## 二、路由策略：模型输出类型决定后端接口

### 2.1 核心约束

文生图不能走 `/chat/completions` 的原因：**请求和响应的格式完全不同**。

| 维度 | 对话 | 文生图 |
|------|------|--------|
| 端点 | `POST /chat/completions` | `POST /images/generations` |
| 请求体 | `{ model, messages, stream: true }` | `{ model, prompt, size }` |
| 响应 | 流式文本（SSE） | 图片 URL 或 base64 |
| 是否支持 stream | 是 | 否（部分模型显式拒绝） |

把文生图的 prompt 塞进 `messages` 调 `/chat/completions`，网关直接拒接；反过来把对话历史传给 `/images/generations` 也一样无效。所以 **在发起 HTTP 请求之前就必须决定走哪个接口**。

### 2.2 路由判断逻辑

```javascript
// main/main.js — chat:send-stream 中：
const isImage = brain.type === 'image' || isImageModel(brain.modelName);
if (isImage) {
  // → POST /images/generations
} else {
  // → POST /chat/completions（流式）
}
```

**优先级：`brain.type`（用户手动标记）> `isImageModel()`（关键词兜底）**

### 2.3 关键词匹配函数

用于兼容早期没有 `type` 字段的数据。检测模型名是否包含已知文生图关键词：

```javascript
function isImageModel(modelName) {
  const imageKeywords = [
    'flux', 'stable-diffusion', 'sdxl', 'dall-e',
    'black-forest-labs', 'stabilityai', 'deep-floyd',
    'pixart', 'latent-consistency', 'playground',
    'kandinsky', 'würstchen', 'cogview', 'glm',
    'wanx', 'taiyi', 'midjourney',
  ];
  const lower = modelName.toLowerCase();
  return imageKeywords.some(keyword => lower.includes(keyword));
}
```

**为什么需要兜底：** 方案切换前已有的大脑数据没有 `type` 字段（默认值为 `"chat"`），如果不做关键词检测，这些老数据配置的文生图大脑会被错误路由到 `/chat/completions`。

### 2.4 Brain.type 字段

```prisma
model Brain {
  id        String   @id @default(uuid())
  name      String
  provider  String
  modelName String
  apiKey    String
  baseUrl   String?
  type      String   @default("chat")  // "chat" | "image"
  createdAt DateTime @default(now())
  roles     Role[]
}
```

`type` 字段在 BrainModal 中通过两个切换按钮设置，用户必须明确选择"对话"或"文生图"才能保存。

## 三、文生图完整流程

```
用户输入 prompt
       │
       ▼
chat:send-stream IPC
       │
       ▼
brain.type === 'image' ?  ─否→  brain.type === 'chat' && isImageModel() ?  ─否→  /chat/completions（文字）
       │                           │
       是                          是
       │                           │
       └───────────┬───────────────┘
                   ▼
      POST {baseUrl}/images/generations
        { model, prompt, size }
                   │
          ┌────────┴────────┐
          ▼                 ▼
       成功               失败
          │                 │
          ▼                 ▼
    下载图片到本地       chat:error
    构造 app-img:// URL
    存 Message(type: image)
          │
          ▼
     chat:done
          │
          ▼
     前端刷新消息列表
     ChatView 渲染 <img>
```

### 3.1 涉及文件

| 文件 | 作用 |
|------|------|
| `main/main.js` | `isImageModel()` 检测；`chat:send-stream` 路由分发；`app-img://` 协议注册 |
| `src/app/components/ChatView.tsx` | 按 `msg.type` 分叉渲染：`text` → Markdown，`image` → `<img>` |
| `src/app/components/BrainModal.tsx` | 模型行末尾的「对话 / 文生图」切换按钮 |
| `prisma/schema.prisma` | Brain.type + Message.type 字段定义 |

## 四、厂商 API 兼容性

### 4.1 格式统一性

目前三个主要厂商均支持 OpenAI 兼容格式的 `/images/generations`：

| 厂商 | baseUrl | model 示例 |
|------|---------|-----------|
| 硅基流动 | `https://api.siliconflow.cn/v1` | `black-forest-labs/FLUX.1-dev`、`Kwai-Kolors/Kolors` |
| 智谱 GLM | `https://open.bigmodel.cn/api/paas/v4` | `glm-image`、`cogview-3-flash`、`cogview-4` |
| 百炼 | `https://dashscope.aliyuncs.com/compatible-mode/v1` | 通义万相系列 |

### 4.2 注意事项

- **智谱模型名**：必须是有效文生图模型（`glm-image` 或 `cogview-*`）。无效模型名走 `/images/generations` 也会失败
- **智谱 1212 错误**：`"当前模型不支持SSE调用方式"`。触发条件：文生图模型被路由到 `/chat/completions` + `stream: true`（即 `isImageModel()` 漏匹配且 `brain.type` 未标记为 image）
- **图片有效期**：API 返回的 URL 通常有时效（智谱 30 天），生产环境应下载到本地

## 五、图片存储方案

### 5.1 目录结构

```
prisma/images/{roleId}/{timestamp}.png    # 开发环境
userData/images/{roleId}/{timestamp}.png   # 生产环境（app.getPath('userData')）
```

### 5.2 Message 记录

```prisma
model Message {
  content   String   // "app-img:///E:/.../image.png"（包含自定义协议头的绝对路径）
  type      String   @default("text")  // "text" | "image"
  // ...
}
```

### 5.3 前端渲染

```jsx
// ChatView.tsx
{msg.type === 'image' ? (
  <img src={msg.content} alt="生成的图片" className="max-h-64 w-auto rounded-lg" />
) : (
  <ReactMarkdown>{msg.content}</ReactMarkdown>
)}
```

## 六、设计原则与边界

### 6.1 原则

1. **路由不靠猜测**——用户配置 API 时清楚模型功能，手动标记最可靠
2. **关键词只做兜底**——不依赖它做主要判断，仅用于兼容老数据
3. **统一界面**——ChatView 处理所有消息类型，渲染逻辑按 Message.type 分叉
4. **本地优先**——图片存本地文件，不依赖 API URL 有效期
5. **Electron 特性**——自定义协议 + 主进程文件操作，渲染进程无文件系统权限

### 6.2 明确不做的事

- 多图生成（`n > 1`）
- 图片删除 / 管理
- 图片编辑 / 放大
- 文生视频或其他多模态输出
- 流式输出（文生图不支持流式）
- 维护已知文生图模型数据库
