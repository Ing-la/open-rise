## 1. 前置知识：JS 基础速查

### 变量声明：`const` vs `let`

```javascript
const x = 5;   // 常量，不可重新赋值（类似 Python 的 x = 5 但约定不改）
let y = 10;    // 变量，可重新赋值
// 在 Python 里你直接写 x = 5，没有 const 的概念
```

### 模块导入：`require()` vs Python `import`

```javascript
const fs = require('fs');      // 类似 Python 的 import os
const { app, ipcMain } = require('electron');
// 解构导入，类似 Python 的 from electron import app, ipcMain
```

### 箭头函数：`() => {}` vs Python `lambda`

```javascript
// JS 箭头函数
const add = (a, b) => { return a + b; };
const addShort = (a, b) => a + b;  // 单行可省略 return

// Python 对应
// add = lambda a, b: a + b
```

### 模板字符串：`` `...${}...` `` vs Python f-string

```javascript
const name = 'Alice';
console.log(`Hello ${name}`);  // 类似 Python 的 f"Hello {name}"
```

### 解构赋值 vs Python 解包

```javascript
const obj = { a: 1, b: 2 };
const { a, b } = obj;     // 类似 Python 的 a, b = obj['a'], obj['b']

const arr = [1, 2, 3];
const [x, y, z] = arr;    // 类似 Python 的 x, y, z = arr
```

### 数组方法速查

| JS 写法 | 类似 Python | 说明 |
|---------|-----------|------|
| `arr.map(fn)` | `map(fn, arr)` 或 `[fn(x) for x in arr]` | 每个元素执行 fn，返回新数组 |
| `arr.filter(fn)` | `filter(fn, arr)` 或 `[x for x in arr if fn(x)]` | 保留 fn 返回 true 的元素 |
| `arr.reduce((sum, x) => sum + x, 0)` | `functools.reduce(lambda sum, x: sum + x, arr, 0)` | 累计计算 |
| `arr.forEach(fn)` | `for x in arr: fn(x)` | 遍历执行，不返回值 |
| `arr.slice(0, -N)` | `arr[:-N]` | 负索引截取（和 Python 一模一样！）|
| `arr.find(fn)` | 没有直接等价，需 `next(x for x in arr if fn(x))` | 返回第一个匹配元素 |
| `arr.some(fn)` | `any(fn(x) for x in arr)` | 是否有任意元素满足条件 |
| `arr.join('\n')` | `'\n'.join(arr)` | 注意 Python 是字符串调 join |

### `??` vs `||` vs Python `or`

```javascript
// ?? — 只在「null 或 undefined」时取默认值（最安全）
const x = null ?? 'default';   // → 'default'
const y = 0 ?? 'default';      // → 0（0 不是 null，所以保留 0）
const z = '' ?? 'default';     // → ''（空字符串不是 null，保留）

// || — 「falsy」时取默认值（falsy = false, 0, '', null, undefined）
const a = '' || 'default';     // → 'default'（空字符串是 falsy）
const b = 0 || 'default';      // → 'default'

// Python or 行为类似 ||，不是 ??
// '' or 'default' → 'default'  ❌ 0 or 'default' → 'default'
```

### ⚠️ `??` 不能和 `||` 混用

```javascript
// 错误语法：JS 禁止混合 ?? 和 ||，会报 SyntaxError
// const x = a ?? b || c;

// 正确做法：加括号明确优先级
const x = (a ?? b) || c;  // 先 ?? 再 ||
const y = a ?? (b || c);  // 先 || 再 ??
```

### 可选链：`?.` vs Python `?.` 不存在

```javascript
// data.choices?.[0]?.message?.content
// 如果 data 为 null/undefined，不报错，返回 undefined
// Python 没有这个语法，得写：
// if data and data.get('choices') and data['choices'][0] ...
```

### Spread 操作符：`...arr`

```javascript
const arr1 = [1, 2, 3];
const arr2 = [...arr1, 4, 5];  // → [1, 2, 3, 4, 5]，类似 Python 的 [*arr1, 4, 5]
```

### `Set` — 类似 Python 的 `set`

```javascript
const set = new Set();
set.add('abc');
set.has('abc');  // true
set.delete('abc');
// 类似 Python 的 set()，但没有字面量语法
```

### `try` / `catch` 无参语法

```javascript
// JS 允许 catch 省略参数（Python 不允许）
try {
  JSON.parse(badData);
} catch {
  // 不关心错误类型，直接跳过
}

// Python 必须写 except Exception:
// try:
//   json.loads(bad_data)
// except Exception:
//   pass
```

**`ipcMain.on` vs `ipcMain.handle`** — 两种完全不同的 IPC 模式：

```
ipcMain.handle('xxx', handler)
  → 前端 ipcRenderer.invoke('xxx')
  → 返回 Promise，前端 await 等待结果

  → 前端 ipcRenderer.send('xxx')
```

对比 Python：


**SSE 协议格式** — LLM 流式 API 按 SSE (Server-Sent Events) 协议返回：


---

```javascript
    for (const line of lines) {
      // 去掉 "data: " 前缀（前 6 个字符），再 trim 掉空白
      const data = line.slice(6).trim();

      // 如果是结束标记，跳过
      if (data === '[DONE]') continue;

      try {
        // 解析 JSON 字符串为 JS 对象
        const json = JSON.parse(data);

        // 安全提取 delta.content（当前 token 的内容）
        // choices[0].delta.content = 本轮输出的文本片段
        // 可选链 ?. 保护中间 null 的情况
        // || '' 将 undefined 转为空字符串
        const delta = json.choices?.[0]?.delta?.content || '';

        if (delta) {
          fullReply += delta;                        // 累计完整回复
          event.sender.send('chat:chunk', { content: delta });  // 推送 token 到前端
        }
      } catch { /* skip malformed SSE */ }
    }
  }
```

**`line.slice(6)`** — 字符串切片，从索引 6 开始（即跳过 `data: ` 这 6 个字符）。Python 是 `line[6:]`。

**`JSON.parse(data)`** — 将 JSON 字符串解析为 JS 对象。相当于 Python 的 `json.loads(data)`。

**`catch { /* skip */ }`** — JS 允许 `catch` 不带参数。Python 必须写 `except Exception:`。

**Fire-and-forget** — 调用异步函数但不 `await` 它。相当于 Python 的：

```python
# Python 里异步函数需要 await 或 asyncio.create_task
asyncio.create_task(check_and_compress(role_id))

# .catch() 相当于 try/except
# 在 Python 里：
async def safe_compress(role_id):
    try:
        await check_and_compress(role_id)
    except Exception as e:
        print(f'Compression failed: {e}')
```

为什么不用 `await`？因为压缩不是用户必须等待的操作。`await` 会阻塞，用户就得等压缩完才看到回复已结束。不 `await` 的话，压缩在后台悄悄进行，回复立刻展示。

## 5. `compressMessages` — LLM 压缩函数

文件：`main/handlers/chat.js`，第 34-72 行

```javascript
// async function = 异步函数，类似 Python async def
// 参数：role = Role 对象（包含 brain, soul, rule, summary）
//       messages = 待压缩的消息数组
async function compressMessages(role, messages) {
  // 解构赋值：从 role 中取出 brain, soul, rule, summary
  const { brain, soul, rule, summary } = role;

  // 如果角色没有绑定大脑，无法调 LLM，保留旧的 summary
  if (!brain) return summary || '';
```

`return summary || ''` — 逻辑或运算符 `||`：
- 如果 `summary` 是假值（null, undefined, 空字符串），返回 `''`
- 否则返回 `summary` 本身

```javascript
  // 将消息数组转换为纯文本格式
```

### 关于 `/ `(`?? summary`) || ''` 的说明

这是一个重要的语法细节。旧版曾写 `data.choices?.[0]?.message?.content ?? summary || ''`，但 **JS 禁止直接混用 `??` 和 `||`**，会报 SyntaxError。

所以必须加括号：

```javascript
// ✅ 正确：(content ?? summary) || ''
// 1. 先 content ?? summary：content 是 null/undefined 时用旧 summary
// 2. 再 || ''：如果上一步结果是 falsy（null/undefined/''）→ 用空字符串

// ❌ 错误：content ?? summary || ''（SyntaxError）
```

逻辑分解：

| 情况 | content | summary | (content ?? summary) \|\| '' |
|------|---------|---------|---------------------------|
| 正常响应 | "新摘要" | 任意值 | "新摘要" |
| content 为 null | null | "旧摘要" | "旧摘要" |
| content 和 summary 都为空 | null | null | '' |
| content 为空字符串 | '' | "旧摘要" | ''（因为 ?? 保留 ''） |

```javascript
      body: JSON.stringify({
        model: brain.modelName,
        max_tokens: SUMMARY_MAX_TOKENS,
        messages: [
          { role: 'system', content: 'You are a conversation summarizer. Produce a concise, information-dense summary.' },
          { role: 'user', content: summaryPrompt },
        ],
      }),
    });

    if (!response.ok) return messages;

    const data = await response.json();
    const summary = data.choices?.[0]?.message?.content;
    if (!summary) return messages;

    console.log(`[AgentCompact] Compressed session ${sessionId} → ${transcriptPath}`);

    // 替换整个消息列表：只保留 system prompt + 摘要
    return [
      { role: 'system', content: messages.find((m) => m.role === 'system')?.content || '' },
      { role: 'user', content: `[会话已压缩，以下为之前的摘要]\n\n${summary}\n\n请继续完成任务。` },
    ];
  } catch (err) {
    console.error('[AgentCompact] Compression failed:', err.message);
    return messages;
  }
}
```

**`messages.find((m) => m.role === 'system')`** — 返回第一个 role 为 'system' 的消息。Python 里是 `next(m for m in messages if m['role'] == 'system')`。

压缩后 Agent 只保留两条消息：
1. 原始的 system prompt
2. 一条包含摘要的 user 消息

### shouldCompact — 触发判断（第 138-140 行）

```javascript
}
```

### 三层压缩流程总结

```
每轮 Agent 循环
    │
    ├── 第 1 层：microCompact（每次都执行）
    │     从后往前找 tool_result
    │     超过 5 轮的 → "[Previous: used {toolName}]"
    │
    └── 第 2 层：shouldCompact（检查 token 总量 > 40000）
            │
            是 → autoCompact
                  ├── 保存完整对话到 .jsonl
                  ├── 调 LLM 生成摘要
                  └── 替换为 [system + 摘要] 两条消息

第 3 层：用户输入 /compact 命令
            → 调用 autoCompact（相同逻辑，手动触发）
```

Agent 循环中调用压缩的代码（`main/handlers/agent.js` 第 149-155 行）：

```javascript
    // Memory compaction
    microCompact(messages);
    if (shouldCompact(messages)) {
      event.sender.send('agent:progress', { sessionId, status: 'compacting', message: '正在压缩记忆...' });
      messages = await autoCompact(messages, sessionId, role.name, brain);
      event.sender.send('agent:progress', { sessionId, status: 'thinking', message: '继续工作...' });
    }
```

注意：Agent 里 `messages` 是局部变量（通过 `let messages = ...` 声明），所以 `autoCompact` 返回的新数组可以直接赋值给 `messages`。

## 10. Prisma Schema 更新

文件：`prisma/schema.prisma`

### Role 模型 — summary 字段

```prisma
model Role {
  id        String    @id @default(uuid())
  name      String
  soul      String
  rule      String
  summary   String?   // ⭐ 长期记忆：LLM 压缩后的历史对话摘要
  avatar    String?
  brain     Brain     @relation(fields: [brainId], references: [id])
  brainId   String
  messages  Message[]
  agentSessions AgentSession[]  // Agent 会话（一对多）
  createdAt DateTime  @default(now())

  @@index([brainId])
}
```

**`String?`** — 问号表示可选（可 null）。对应 SQL 的 `TEXT`（无 NOT NULL）。

**关系字段**: `agentSessions AgentSession[]` — 一个角色可以有多个 Agent 会话。

### Agent 相关模型

```prisma
model AgentSession {
  id        String   @id @default(uuid())
  roleId    String
  title     String
  status    String   @default("active")
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt  // 每次更新自动修改时间戳

  role      Role      @relation(fields: [roleId], references: [id])
  messages  AgentMessage[]

  @@index([roleId])
}
```

**`@updatedAt`** — Prisma 自动维护的时间戳，每次更新记录时自动修改。没有 Python/SQLAlchemy 的直接等价物，得手动维护。

```prisma
model AgentMessage {
  id        String   @id @default(uuid())
  sessionId String
  role      String   // "user" / "assistant" / "tool_result"
  content   String
  type      String   // "text" / "tool_call" / "tool_result"
  toolName  String?  // 工具名（仅 tool_result 时）
  toolId    String?  // OpenAI 工具调用 ID（关联 tool_call）
  createdAt DateTime @default(now())

  session AgentSession @relation(fields: [sessionId], references: [id])

  @@index([sessionId])
}
```

对比 Message 表和 AgentMessage 表：

| 维度 | Message（聊天） | AgentMessage（Agent） |
|------|----------------|---------------------|
| 外键 | `roleId` → Role | `sessionId` → AgentSession |
| 字段 | sender, content, type | role, content, type, toolName, toolId |
| 压缩方式 | 删除旧消息 + summary 字段 | 替换为压缩摘要消息 |
| 图片支持 | type='image' | 不支持 |

### 聊天记忆压缩

```

          ├── streamChatResponse()
          │     ├── fetch(stream:true) → SSE 解析
          │     ├── 逐 chat:chunk 推送前端
          │     ├── 流结束 → 存完整回复
          │     └── checkAndCompress（异步 fire-and-forget）
          │
          └── checkAndCompress
                │
                ├── 条件：count > 10 且 chars > 20000？
                │
                └── 执行：
                      ├── compressMessages → LLM 生成新 summary
                      ├── formatArchiveMarkdown → 格式化
                      ├── getArchiveDir → 写文件
                      ├── prisma.role.update → 更新 summary
                      └── prisma.message.deleteMany → 删旧消息
```

### Agent 记忆压缩

```
Agent 每轮循环（最多 30 次迭代）
    │
    ├── microCompact → 替换旧 tool_result 为占位符
    │
    ├── shouldCompact → token > 40000？
    │     └── autoCompact → LLM 摘要 → 替换消息列表

    ├── 调 LLM API（带工具）
    ├── 解析响应 → tool_call? → 执行工具
```
