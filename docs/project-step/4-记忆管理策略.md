# 4. 记忆管理策略

## 设计原则

- **短期记忆 = Message 表全部消息**，每次对话全部送入上下文
- **阈值自控**：消息总量超过 token 阈值时触发压缩，不设硬性条数上限
- **LLM 主导**：压缩时 LLM 自行决定保留 vs 舍弃
- **历史可追溯**：归档到 markdown 磁盘文件，信息不丢失

---

## 工作流程

### 正常对话

```
用户输入
  ↓
查 Role（含 summary + brain）
  ↓
拼 system prompt = soul + rule + summary
  ↓
拼 messages = Message 表全部消息（受阈值控制，通常 ≤ 30 条）
  ↓
调用 LLM API
  ↓
存 assistant 回复到 Message 表
  ↓
返回回复给用户
  ↓
异步压缩检查：totalChars / 4 > 3000 tokens？
```

### 压缩触发

```
totalChars > 20000 chars？
  ↓ 是
保留最近 10 条，其余全部作为 toArchive
toArchive + 旧 summary → LLM → 新 summary（max_tokens: 2048）
toArchive → 写入 prisma/archives/{name}-{ts}.md
更新 Role.summary = 新 summary
删除已归档的早期消息
  ↓
结果：10 条消息 + 新 summary，继续对话直到再次超阈值
```
---

## 压缩提示词（Prompt）

压缩质量完全取决于 LLM 收到的指令。当前使用的提示词如下：

### 完整 Prompt

```text
System:
You are a conversation summarizer. Produce a concise summary preserving
key facts, decisions, preferences, and user context. Output ONLY the
summary, no preamble.

User:
Previous summary: {旧 summary 或 (none)}

Character personality: {soul}
Character rules: {rule}

Conversation to summarize:
[{sender}]: {message content}
[{sender}]: {message content}
...

Concise summary:
```

### 各组件作用

| 组件 | 作用 |
|------|------|
| **System prompt** | 设定任务角色，约束输出格式（纯文本，无前缀说明） |
| **Previous summary** | 传递旧 summary 作为上下文，新 summary 在其基础上增量更新，而非从头重写 |
| **Character personality / rules** | 提供角色的灵魂和规则，让 LLM 知道哪些对话内容对角色的连贯性重要 |
| **Conversation to summarize** | 待压缩的原始对话全文，按 `[sender]: content` 格式逐条排列 |
| **Concise summary 指令** | 最终提示 LLM 输出，结合 system prompt 的格式约束 |

### 设计要点

1. **增量而非全量**：传入 Previous summary 是最关键的设计。压缩不是从零总结，而是在旧 summary 基础上补充/更新新对话的内容。保证长期信息不丢失。
2. **角色感知**：soul + rule 作为上下文信息传递给 LLM，让它能判断哪些对话内容与角色设定相关、值得保留。
3. **关注点明确**：明确要求保留 key facts（关键事实）、decisions（决策）、preferences（偏好）、user context（用户信息）。这些是跨对话轮次最有价值的信息。
4. **无格式噪音**：`Output ONLY the summary, no preamble` 确保输出是纯文本，直接存入 `Role.summary` 字段，无需后处理。

### 信息传递链路

```text
对话消息 → [LLM 压缩] → summary（纯文本） → 存入 Role.summary
                                            ↓
                             下次对话时拼入 system prompt
                                            ↓
                          soul + rule + summary 作为上下文
                                            ↓
                                     LLM 生成回答
```

旧 summary 始终在循环中：被读出 → 参与压缩（作为 Previous summary）→ 被更新 → 存回 → 再次被读出。

### 注意事项

- 压缩 prompt 当前硬编码在 `compressMessages` 函数中（`main/main.js:280-286`）
- 如果要调整压缩策略（如侧重保留对话风格、增加语言一致性要求等），改此处的 prompt 即可
- 修改后建议观察几轮压缩输出质量，确保新 summary 保留了需要的信息

## 关键参数

| 参数 | 值 | 说明 |
|------|----|------|
| `COMPRESS_CHAR_THRESHOLD` | 20000 chars | 短期记忆字符数超此值触发压缩 |
| `SUMMARY_MAX_TOKENS` | 2048 | 压缩输出的 summary 长度上限 |
| `SHORT_TERM_KEEP` | 10 条 | 压缩后保留的最新消息条数 |

这三个参数在 `main/main.js` 顶部 Compression 段以常量定义，是压缩逻辑的唯一调优点。

## 存储

- **短期记忆**：`Message` 表，每次对话全部送入上下文
- **长期记忆**：`Role.summary` 字段（TEXT），始终在 system prompt 中
- **历史归档**：`prisma/archives/*.md`（开发）或 `userData/archives/*.md`（生产），纯文本 markdown，无外部依赖

---

## 与旧方案对比

| 维度 | 旧方案（Memory 表） | 新方案（summary 字段） |
|------|-------------------|----------------------|
| 存储 | Memory 表 + 索引 | Role.summary 字段 |
| 检索 | read_memory 工具按标题查 | summary 始终在 system prompt |
| 短期上限 | 硬编码 take: 30 | 无硬上限，阈值自控 |
| 压缩策略 | 取最早 20 条 | 保留最近 10 条，其余全部 |
| 历史 | 覆盖写入 Memory 表 | markdown 磁盘归档 |
| 复杂度 | 高（3 层 + IPC + 工具函数） | 低（自动异步压缩，零 IPC） |

## 上下文预算

| 组件 | 约占比 |
|------|--------|
| system prompt（soul + rule + summary） | ~1-3K tokens |
| 短期记忆（压缩前最多 20000 chars） | ~10-13K tokens |
| 当前输入 | ~0.5K tokens |
| **合计** | **~15K tokens** |
| 128K 上下文 | 剩余 ~113K，充裕 |

短期记忆在 20000 chars 处压缩 → 回落到 10 条（约 2000-4000 chars）→ 再次增长 → 再次压缩。128K 上下文下这个阈值可以很宽松。
