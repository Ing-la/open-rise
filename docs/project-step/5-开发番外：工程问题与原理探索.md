## 一、CLI 与 TUI：另一种技术路线

### 1.1 从 Electron 到 CLI 的转变

当你从 **Electron + React** 转向 **CLI** 时，实际上是从"视觉驱动"转向了"逻辑与信息密度驱动"。Claude Code 便是 CLI 路线的代表——它不再使用 Electron 打包整个浏览器，而是直接运行在系统终端中。

**技术栈对比：**

| 维度 | Electron 路线 (GUI) | CLI 路线 (TUI) |
|------|--------------------|----------------|
| **渲染引擎** | Chromium（内存消耗大） | 系统终端（极轻量） |
| **UI 框架** | React / Next.js | **Ink** / Clack / Blessed |
| **部署方式** | `.exe` / `.dmg` 安装包 | `npm install -g` 命令行工具 |
| **视觉上限** | 极高（图片、动画、3D） | 低（字符、颜色、基本图形） |
| **开发效率** | 需处理复杂的 CSS 和布局 | 专注于文字流和命令逻辑 |

### 1.2 Ink：命令行里的 React

Claude Code 使用了名为 **Ink** 的库，这是一个令人惊讶的事实：它允许你使用 **React 的组件化思想**来构建命令行界面（TUI）。

- **原理：** 你依然可以写 `<Box>`、`<Text>`、`<Spinner>` 这样的组件
- **渲染机制：** Ink 最终不是渲染成 HTML 标签，而是渲染成终端能识别的 **ANSI 转义字符**
- **布局系统：** 使用 Flexbox 弹性盒子布局，确保窗口缩放时信息不会错乱

### 1.3 TUI 设计的核心

即使没有图形界面，TUI 也需要设计，其核心是：

- **布局逻辑：** 如何在有限的字符行数里排列信息
- **信息密度：** 文字的颜色（ANSI Colors）、粗细和缩进成为唯一的"设计语言"
- **反馈设计：** Spinner 的旋转频率、操作成功后的勾号——这些都是 TUI 设计的一部分

### 1.4 终端中的图像

关于终端中显示图片的问题：

- **ASCII / Unicode 艺术：** 使用特殊 Unicode 字符拼出图形
- **六角格（Sixel）与图形协议：** 部分现代终端（Kitty、iTerm2）支持 Sixel 协议，可将真实图像数据编码后在终端直接"画"出来
- **SVG 转换：** 很多 CLI 工具在后台将复杂图形转换成简化的 ASCII 灰度图显示

## 二、LLM API 与 ReAct 框架

### 2.1 三种 API 模式

在 2026 年，LLM API 可分为三类：

#### 模式一："指挥家"模式（标准 Chat API）——不包含 ReAct

最常见的接口（如 Claude Messages API、OpenAI Chat API、DeepSeek Chat API）。

- **逻辑：** 模型只是"指挥家"，它告诉你想调用工具，但不会自己去执行
- **循环在哪里？** **循环逻辑必须写在你的本地代码里。** 你需要拦截模型输出，发现它要调工具，本地执行后再把结果喂回 API
- **官方文档：** 这类 API 强调 **"Tool Use (Function Calling)"**，明确说明由调用方负责提交 `tool_results`

> **DeepSeek V4 Flash 和 Pro** 本质上都是单次通行模型。它们的核心任务是给定一段输入，预测下一段输出，不会自发地循环思考。如果你驱动的模型说"我需要调用搜索工具"，它会停在原地，等你的本地代码去执行。

#### 模式二："项目经理"模式（Agent 平台级 API）——内置 ReAct

一些厂商推出了自带"托管循环"的接口：

- **Google Gemini Enterprise (2026)：** Agent Platform 支持自主推理和多步工作流，发一个请求，Google 后台接管整个 ReAct 循环
- **OpenAI (2026)：** 将 Assistants API 迁移到 Responses API 和 Conversations API，支持服务器端多步工具调用
- **文档特征：** 你会看到 **"Run"**、**"Thread"** 或 **"Managed Orchestration"** 等字眼

#### 模式三："深度思考"模式（推理模型）——内部 CoT，非 ReAct

像 **OpenAI o1/o3**、**DeepSeek-R1** 这类模型，调用时它们确实会"思考很久"才输出。

- **这算 ReAct 吗？** **不算。** 这叫"内部思维链"（Internal Chain of Thought）
- **原理：** 模型在输出第一个字之前，在其内部进行了海量的递归推理。但这是纯逻辑推导，**它并没有在这个过程中去访问外网或数据库**
- **本质：** 依然是单次推理，只是这单次推理变得非常长、非常深

### 2.2 对 ZOE 开发的实际建议

坚持在本地写循环逻辑，理由：

1. **控制权：** 把 ReAct 循环交给官方 API，就失去了对每一步推理的"审查权"
2. **调试透明度：** 本地写 ReAct（Thought → Action → Observation）能清楚看到 Agent 在哪个环节卡住
3. **成本优化：** 官方托管 Agent 通常产生更高的"管理费"，本地循环可以更精细地控制何时停止、何时切换模型

## 三、SVG 技术详解

### 3.1 SVG 是什么？

**SVG (Scalable Vector Graphics)**，即可缩放矢量图形。它是 Web 开发中唯一的"一等公民"图形格式。

### 3.2 技术栈：它是代码，也是图片

SVG 完全基于 **XML**，处于一个神奇的技术交叉点：

- **HTML/XML：** SVG 标签（`<rect>`、`<circle>`、`<path>`）可直接写在 HTML 中
- **CSS：** 可用 CSS 设置填充色（`fill`）、描边（`stroke`）或添加动画
- **JavaScript：** 可动态修改 SVG 的坐标和属性，实现复杂的交互逻辑

### 3.3 底层原理：坐标与数学

不同于位图（JPG/PNG）通过像素点拼凑图像，SVG 使用**几何描述**：

- **矢量指令：** 画一个圆不存储成千上万的像素点，而是存储圆心坐标、半径和填充色：`C = (x, y), r = 10, fill = red`
- **贝塞尔曲线 (Bézier Curve)：** 通过控制点和锚点用数学公式描述复杂曲线。无论放大多少倍，边缘永远丝滑，不会出现"马赛克"

### 3.4 解决的问题

| 解决的问题 | 详细描述 |
|-----------|---------|
| **无限缩放** | 无论手机屏幕还是 4K 显示器，甚至打印成海报，永远不失真 |
| **性能优化** | 简单几何图形（图标、Logo）的体积远小于 PNG |
| **可搜索性** | SVG 内部文字是真正的文本，搜索引擎可抓取，屏幕阅读器可朗读 |
| **动态交互** | SVG 是 DOM 的一部分，可实现"鼠标悬停线条颤抖"等复杂交互 |

### 3.5 SVG 在 ZOE 项目中的妙用

- **"颤抖线条"效果：** 给 SVG 的 `<path>` 添加 `feTurbulence` 滤镜，让静态线条产生随机位移，实现炭笔手绘的动态感
- **轻量级头像：** 每个 Agent 头像用几十行 SVG 描述，随机改变路径参数即可生成无限个独一无二的头像
- **CLI 兼容：** 即使做命令行版本，现代终端也支持通过特定协议显示 SVG 渲染的图形
